## 关键风险点（分析）
- `SSE` 路由若未严格做“项目归属校验”，有跨用户数据泄露风险。  
- 限流为进程内内存实现，多实例部署会失效，建议迁移 Redis/网关限流。  
- 环境变量缺失时有占位逻辑，生产建议“缺关键配置直接启动失败”。  
- 成本统计目前偏业务估算，建议增加可对账的调用明细。  
- 多厂商 API 变更风险较高，建议建立契约测试与统一降级策略。

数据库由五张核心表组成，围绕 `projects` 表构建。其中有一个需要特别解释的**双写设计**：场景数据同时存在于 `projects.storyboard`（JSONB）和 `scenes` 表中，稍后会详细说明原因。---

## 一、五张表的职责和设计决策

### 1. `user_profiles` — 用户元数据

Supabase Auth 管理登录（`auth.users`），`user_profiles` 在应用层扩展用户信息。通过一个数据库触发器，在新用户注册时自动创建对应记录，无需应用代码干预。

### 2. `user_api_keys` — 加密的 API 密钥

每个用户、每个供应商（google/openai/anthropic 等）存储一条记录，密钥在写入前由 `lib/encryption.ts` 用 AES-256-GCM 加密，数据库中只存密文。

### 3. `projects` — 核心业务表（含 JSONB 管道数据）

这里有个关键设计决策：`style_dna`、`research_report`、`narrative_map`、`script`、`storyboard` 这五个字段用 JSONB 而不是独立的规范化表。原因是这些数据结构复杂且频繁变化（每次管道步骤都在覆写整个对象），JSONB 允许原子更新整个结构，避免跨多表的事务。

### 4. `scenes` — 场景资产表（双写设计）

这是最需要理解的设计点。场景数据同时存在于两处：

- `projects.storyboard`（JSONB）：存储完整的场景结构，由步骤 7 生成脚本时整体写入
- `scenes` 表：为每个场景单独存储媒体资产 URL 和状态

为什么要双写？因为步骤 9/10/11 需要**逐场景并发更新** `keyframe_url`、`video_url`、`audio_url` 和 `status`，如果每次都更新整个 `storyboard` JSONB，并发写入时会互相覆盖。步骤 12 渲染时再将两者合并读取。

### 5. `pipeline_events` — 实时进度日志

每个管道步骤的开始/完成/失败都写一条记录，SSE 接口订阅这张表的 Realtime 变更来推送进度到前端。

## 二、现有 Migration 的问题

原始 `001_initial.sql` 有以下需要修复的地方：

1. `user_api_keys` 外键指向 `user_profiles(id)`，但触发器创建 profile 是异步的，可能导致外键约束失败
2. 缺少配额查询所需的复合索引
3. `pipeline_events` 的 RLS INSERT 策略过于严格，Inngest（服务端）用 service role 写入时绕过 RLS，但策略逻辑有误
4. 缺少 `storage` bucket 的配置
5. `supabase_realtime` publication 添加方式在新版本有变化

## 四、部署步骤

**方法一：通过 Supabase CLI（推荐）**

```bash
# 1. 安装 CLI
brew install supabase/tap/supabase
supabase --version

# 2. 登录并关联项目
supabase login
supabase link --project-ref axateisbsyensaoqzhat


# 4. 推送到数据库
supabase db push
```

**方法二：Supabase Dashboard SQL 编辑器**

打开 Dashboard → SQL Editor，将文件内容粘贴执行。如果之前已执行过旧版 migration，需要先执行以下清理命令：

```sql
-- 清理旧表（仅开发环境！）
drop table if exists pipeline_events cascade;
drop table if exists scenes cascade;
drop table if exists projects cascade;
drop table if exists user_api_keys cascade;
drop table if exists user_profiles cascade;
drop trigger if exists on_auth_user_created on auth.users;
drop function if exists public.handle_new_user();
```

**验证是否部署成功**

```bash
# 调用健康检查接口，检查 supabase 是否返回 ok
curl https://你的域名.com/api/health
```

返回的 `checks.supabase.status` 为 `ok` 表示数据库连接正常。

## 五、需要同步修改的应用代码

数据库部署后，还有两处代码需要对齐 schema 变化：

**`services/quota.ts` 中的类型转换**（目前存在隐式类型转换风险）：

```ts
// 修改前（类型不安全）
const costToday = (costTodayResult.data ?? []).reduce(
  (sum, p) => sum + Number((p as { total_cost_usd: number }).total_cost_usd ?? 0), 0
);

// 修改后（直接使用 Supabase 推断类型）
const costToday = (costTodayResult.data ?? []).reduce(
  (sum, p) => sum + (Number(p.total_cost_usd) || 0), 0
);
```

**`services/observability.ts` 中的成本精度**（schema 已改为 `numeric(10,6)`，代码侧兼容）：

原来 `total_cost_usd numeric(10, 4)` 精度不够，改为 `numeric(10, 6)` 后可以精确记录到微分（0.000001 USD），Inngest 管道里的成本累加逻辑不需要改。
